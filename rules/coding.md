# Coding Rules

## 通用原则

- **最小改动**：只修改必要的代码，不做推测性优化
- **避免无关修改**：不混合功能开发和重构
- **可读性优先**：代码清晰度优先于性能简写
- **风格一致**：跟随现有代码风格，不引入新风格

---

## Markdown 规则

- **标题**：使用 ATX 风格（# ## ###），前后各空一行
- **列表**：使用 `-` 而非 `*`，同级缩进对齐
- **代码块**：标明语言（``` + 语言名）
- **表格**：使用 GFM 格式
- **加粗**：使用 `**` 而非 `__`
- **文件引用**：使用反引号包裹路径
- **行宽**：每行不超过 100 个字符
- **分隔线**：使用 `---`，前后各空一行

---

## HTML / CSS 规则

- **语义化标签**：优先使用语义标签
- **样式分离**：CSS 写入 `<style>` 或独立文件，避免行内样式
- **缩进**：2 空格
- **ID/Class 命名**：kebab-case，如 `product-gallery`
- **编码**：UTF-8

---

## JavaScript 规则

- **变量声明**：优先 `const`，需重新赋值用 `let`，不用 `var`
- **命名**：变量/函数 camelCase，类 PascalCase
- **字符串**：单引号 `''`，模板字符串用反引号
- **错误处理**：使用 try/catch，避免静默失败
- **异步**：优先 `async/await` 而非裸 Promise
- **分号**：跟随项目现有风格

---

## JSON 规则

- **缩进**：2 空格
- **键名**：双引号包裹
- **尾逗号**：不允许
- **注释**：JSON 不支持注释，需说明时用配套 .md 文件

---

## 文件组织

- **单一职责**：每个文件只做一件事
- **命名**：kebab-case（`architecture-notes.md`），英文小写
- **编码**：UTF-8 无 BOM
- **行尾**：跟随现有项目风格

---

## 注释规范

- **少而精**：只在逻辑复杂处加注释
- **中文文档**：面向用户的文档用中文；代码注释用英文
- **调试代码**：提交前删除所有临时调试代码

---

## 提交前检查清单

- [ ] 改动最小？（没有顺手改无关代码）
- [ ] 风格一致？（跟随现有代码风格）
- [ ] 无死代码？（无残留调试代码或注释块）
- [ ] 可读性高？（变量/函数命名清晰）
- [ ] 无重复逻辑？（无拷贝粘贴重复代码）
