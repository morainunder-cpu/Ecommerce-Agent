## Refactor Workflow

### Objective

Improve readability or maintainability without changing behavior.

### Before Refactoring

Identify:

- Current responsibility
    
- Existing dependencies
    
- Public interfaces
    
- Possible compatibility risks
    

### Refactoring Principles

Prefer:

- Small commits
    
- Small functions
    
- Remove duplication
    
- Better naming
    
- Simpler structure
    

Do not:

- Mix feature development with refactoring
    
- Change business logic unintentionally
    
- Rewrite working code without clear benefit
    

### Verification

Confirm:

- Public behavior unchanged
    
- Tests still pass
    
- Interfaces remain compatible
    

### Output

Always provide:

Current Structure

↓

Problems

↓

Refactor Plan

↓

Changes

↓

Compatibility

↓

Verification

↓

Future Improvements
